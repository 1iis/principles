# Boring is good
> Intelligible before intelligent

Boring is the goal.

Shallow abstractions, flat code, clear signatures, short docstrings, boring interfaces. “This input gives you that output” is often all one should need to know. If we have to explain the structure too much, the structure may be too convoluted.

Boring means usable. Boring means our brain can read it in six months. It means the next person, including future-us, can understand the system without requiring an oracle to narrate it back.

We have little faith in a software process that requires LLMs for humans to understand the codebase. The operative concept is not intelligent tools but [intelligible code][intelligible-systems]. The latter exists; the former is hope.

Simple is hard. Complexity sneaks in through “just one more abstraction,” “just one more option,” “just one more layer.” We have to actively say no, and walk back when we must. This is why [composable boundaries][composable-boundaries] matter.

A solution that requires less context to understand is usually a better solution. Ultimately, boring code serves [context discipline][context-discipline].
